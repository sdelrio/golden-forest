# D&D Tooling: Boneyard + XML Character Data

**Date:** 2024 (approximate)
**Status:** Active
**Decision:** Use boneyard-js for skeletons, XML for character data

## Context
The D&D tools need to display character data (111 characters) with polished loading states and a consistent visual language.

## Choices

### Skeleton Loading: boneyard-js v1.9
- Manual JSON bones in `src/bones/` (no CLI — Docusaurus uses webpack/Rspack, not Vite)
- Registry client module at `src/bones/registry.js` syncs dark mode via MutationObserver
- `initialBones` prop bypasses registry timing issues
- Critical: children must have explicit height (overlay uses `position: absolute; inset: 0`)

### Character Data: XML Files
- 111 character XML files in `static/fg/chars/`
- Index generated by `scripts/build-char-index.py` → `static/fg/chars/index.json`
- Rebuild with `task chars:buildindex`
- Schema: `{ filename, name, race, classes: string[], level: int|null }`
- Consumed by `CharSearch` component (responsive card grid)

### Campaign Data: JSON (Planned)
- `static/fg/campaign/` for sessions, encounters, deaths, achievements
- Not yet implemented — all DnD features are NOT STARTED

## Gotchas
- Boneyard checks `.dark` class, but Docusaurus uses `data-theme='dark'` — registry syncs them
- `registerBones()` must be called in registry (importing JSON isn't enough)
- XML parsing happens client-side via `BrowserOnly` wrappers
